# v1.1.0 Beta-to-GA Transition Guide

**Purpose**: Step-by-step guide for promoting v1.1.0-beta to v1.1.0 GA release
**Audience**: Release Manager, Engineering Team
**Timeline**: February 13-15, 2026 (after 30-day beta period)
**Prerequisites**: Beta testing complete, GA acceptance criteria met

---

## Overview

This guide covers the technical and process steps for transitioning from v1.1.0-beta to v1.1.0 GA (General Availability). The transition involves validation, versioning, testing, documentation updates, and deployment.

**Key Principle**: GA release must be **identical** to the final beta build that passed all acceptance criteria. No code changes between final beta and GA except version number.

---

## Pre-Transition Validation (February 13)

### Step 1: Verify GA Acceptance Criteria

**Time Required**: 2-4 hours

All CRITICAL criteria must pass (see `v1.1.0-ga-acceptance-criteria.md`):

```bash
# C1: Zero Privacy Violations
cd .claude/releases/beta-data/
for export in *.json; do
  echo "Checking $export..."
  jq '.events[] | select(.command != null or .query != null)' "$export"
done
# Expected: NO OUTPUT (zero PII found)

# C2: Pass Rate Target
jq '[.events[] | select(.event_type == "command_generation")] |
    (map(select(.success == true)) | length) / length * 100' \
    beta-telemetry-aggregate.json
# Expected: ‚â•80%

# C3: Zero P0 Bugs
gh issue list --label "P0" --label "v1.1.0-beta" --state open
# Expected: 0 issues

# C4: Installation Success
jq '[.installation_results[]] |
    (map(select(.success == true)) | length) / length * 100' \
    beta-results.json
# Expected: ‚â•95%

# C5: Safety Validation
grep -i "privacy" beta-testing-notes.md
# Expected: ZERO privacy violations documented

# C6: Documentation Accuracy
diff docs/TELEMETRY.md .claude/releases/docs-review/TELEMETRY.md
# Expected: No critical inaccuracies

# C7: User Satisfaction
jq '.survey_responses | map(.overall_satisfaction) | add / length' \
    beta-survey-results.json
# Expected: ‚â•4.0/5.0

# C8: Telemetry Backend
curl -f https://telemetry.caro.sh/health
# Expected: HTTP 200 OK
```

**Decision Point**: If ANY CRITICAL criterion fails ‚Üí DELAY GA, fix issue, repeat validation

**If ALL pass**: Proceed to Step 2

### Step 2: Document Beta Findings

**Time Required**: 1 hour

Create final beta report summarizing:

```bash
cd .claude/releases/
cp v1.1.0-beta-analysis-template.md v1.1.0-beta-final-report.md
```

Fill in:
1. **Quantitative Results**: Pass rates, installation success, satisfaction scores
2. **Qualitative Feedback**: Key themes from tester feedback
3. **Issues Found**: P0-P3 bugs with resolution status
4. **Lessons Learned**: What worked well, what didn't
5. **GA Readiness**: Confidence level and rationale

Commit report:
```bash
git add v1.1.0-beta-final-report.md
git commit -m "docs(release): Final beta testing report for v1.1.0"
git push origin release-planning/v1.1.0
```

### Step 3: Final Go/No-Go Meeting

**Time Required**: 1 hour
**Attendees**: Release Manager, Engineering Lead, Product Owner

**Agenda**:
1. Review GA acceptance criteria results (15 min)
2. Review beta final report (15 min)
3. Discuss any HIGH-severity waivers needed (15 min)

**Outcomes**:
- **GO**: Document decision in `v1.1.0-beta-decision-record.md`, proceed to transition
- **NO-GO**: Document blockers, create remediation plan, set new GA date

## Transition Steps (February 14)

### Step 4: Create GA Release Branch

**Time Required**: 15 minutes

```bash
# Ensure you're on the validated beta commit
git checkout main
git pull origin main

# Identify the exact beta commit to promote
BETA_COMMIT=$(git log --oneline --grep="v1.1.0-beta" -1 | cut -d' ' -f1)
echo "Promoting beta commit: $BETA_COMMIT"

# Create GA release branch from that commit
git checkout -b release/v1.1.0 $BETA_COMMIT

# Verify we're on the right commit
git log -1 --oneline
```

**Critical**: Do NOT merge any other changes. GA must be identical to tested beta.

---

### Step 5: Update Version Numbers

**Time Required**: 30 minutes

Update version from `1.1.0-beta` to `1.1.0` in:

```bash
# 1. Cargo.toml
sed -i '' 's/version = "1.1.0-beta"/version = "1.1.0"/' Cargo.toml

# 2. Cargo.lock (regenerate)
cargo build --release

# 3. Version constant
sed -i '' 's/const VERSION: &str = "1.1.0-beta"/const VERSION: &str = "1.1.0"/' src/version.rs

# 4. README.md
sed -i '' 's/v1.1.0-beta/v1.1.0/g' README.md

# 5. CHANGELOG.md
cat > CHANGELOG_ENTRY.md <<'EOF'
## [1.1.0] - 2026-02-15

### Promoted from Beta
This release promotes v1.1.0-beta to General Availability after successful 30-day beta testing period with 5 beta testers.

**Beta Testing Results**:
- Pass Rate: XX% (target: ‚â•80%)
- Installation Success: XX% (target: ‚â•95%)
- User Satisfaction: X.X/5.0 (target: ‚â•4.0)
- Privacy Violations: 0 (target: 0)
- P0 Bugs: 0 (target: 0)

### Added
[Copy from v1.1.0-beta section]

### Changed
[Copy from v1.1.0-beta section]

### Fixed
[List any bugs fixed during beta]

EOF

# Prepend to CHANGELOG.md
cat CHANGELOG_ENTRY.md CHANGELOG.md > CHANGELOG_NEW.md
mv CHANGELOG_NEW.md CHANGELOG.md
rm CHANGELOG_ENTRY.md

# Verify changes
git diff
```

**Review**: Carefully check all diffs before committing.

```bash
git add Cargo.toml Cargo.lock src/version.rs README.md CHANGELOG.md
git commit -m "chore(release): Bump version to 1.1.0 GA

Promotes v1.1.0-beta to General Availability after successful beta testing.

Beta Results:
- Pass Rate: XX%
- Installation: XX%
- Satisfaction: X.X/5.0
- Privacy: 0 violations
- P0 Bugs: 0"

git push origin release/v1.1.0
```

### Step 6: Build GA Release Binaries

**Time Required**: 1-2 hours (including CI/CD)

```bash
# Tag the GA release
git tag -a v1.1.0 -m "Release v1.1.0 GA

Promotes v1.1.0-beta to General Availability.

Changes since beta: Version number only (no code changes)

Beta Testing Summary:
- 5 beta testers, 30-day testing period
- XX% command generation success rate
- XX% installation success rate
- X.X/5.0 user satisfaction
- 0 privacy violations
- 0 P0 bugs

See CHANGELOG.md for full details."

git push origin v1.1.0

# Trigger CI/CD build (if automated)
# Or build manually:
cargo clean
cargo build --release --locked

# Verify binary version
./target/release/caro --version
# Expected: caro 1.1.0

# Test binary on clean system
docker run --rm -it ubuntu:22.04 bash
# (inside container) Download and run install.sh
# (inside container) Verify: caro --version shows 1.1.0
```

**Build Matrix**:
- ‚úÖ macOS aarch64 (Apple Silicon)
- ‚úÖ macOS x86_64 (Intel)
- ‚úÖ Linux x86_64 (GNU)
- ‚úÖ Linux ARM64

**Quality Gates**:
1. All binaries build successfully
2. All binaries report correct version (1.1.0)
3. Smoke test: `caro "list all files"` succeeds on each platform
4. File sizes reasonable (< 50MB per binary)

### Step 7: Publish to crates.io

**Time Required**: 30 minutes

```bash
# Verify you're on the v1.1.0 tag
git checkout v1.1.0

# Dry run
cargo publish --dry-run

# Review output carefully:
# - Version is 1.1.0 (not 1.1.0-beta)
# - No uncommitted changes
# - All tests pass
# - Documentation builds

# Publish (IRREVERSIBLE)
cargo publish

# Verify publication
sleep 60  # Wait for crates.io to index
cargo search caro
# Expected: caro = "1.1.0"

# Test installation from crates.io
cargo install caro
~/.cargo/bin/caro --version
# Expected: caro 1.1.0
```

**Rollback Note**: crates.io does NOT support unpublishing. Once published, v1.1.0 is permanent. Only proceed if 100% confident.

## Rollback Procedures

### If Issues Found BEFORE Publishing to crates.io

**Severity: P0 Bug Found**

1. **STOP immediately** - Do not publish to crates.io
2. Document the issue in GitHub
3. Assess impact and fix timeline
4. Options:
   - **Quick fix (< 4 hours)**: Fix, test, proceed with GA same day
   - **Longer fix (4-24 hours)**: Delay GA by 1 day, fix, re-test, proceed
   - **Major fix (> 1 day)**: Cancel GA, create v1.1.1-beta for testing

### If Issues Found AFTER Publishing to crates.io

**Severity: P0 Bug Found**

1. **Immediate Actions** (< 1 hour):
   ```bash
   # 1. Create GitHub issue
   gh issue create --label "P0" --label "v1.1.0" \
     --title "P0: [Description]" \
     --body "[Details, impact, reproduction]"

   # 2. Add warning to GitHub release
   gh release edit v1.1.0 --notes-file <(cat <<EOF
   ‚ö†Ô∏è **KNOWN ISSUE**: [Description]

   [Original release notes]
   EOF
   )

   # 3. Add warning to README
   sed -i '1i\> ‚ö†Ô∏è **Known Issue in v1.1.0**: [Description]. Fix coming in v1.1.1. See [issue link].\n' README.md
   git commit -am "docs: Add known issue warning for v1.1.0"
   git push origin main
   ```

2. **Hotfix Process** (< 24 hours):
   ```bash
   # Create hotfix branch from v1.1.0 tag
   git checkout -b hotfix/v1.1.1 v1.1.0

   # Apply minimal fix
   # ... fix code ...

   # Test thoroughly
   cargo test
   cargo build --release
   # ... smoke tests ...

   # Bump to v1.1.1
   sed -i '' 's/version = "1.1.0"/version = "1.1.1"/' Cargo.toml
   git commit -am "fix: [Description] (fixes #XXX)"

   # Tag and publish
   git tag v1.1.1
   cargo publish
   gh release create v1.1.1 --notes "Hotfix for v1.1.0 issue #XXX"
   ```

3. **Communication**:
   - Post update in GitHub Discussions
   - Update social media posts
   - Email users who downloaded v1.1.0 (if possible)
   - Update documentation

### If Telemetry Backend Fails

**Severity: Backend Unavailable**

1. **Immediate Mitigation** (< 15 minutes):
   ```bash
   # Telemetry is opt-in and non-blocking, so caro continues working
   # But users may see upload errors

   # 1. Check backend health
   curl https://telemetry.caro.sh/health

   # 2. If down, check staging
   curl https://staging.telemetry.caro.sh/health

   # 3. If staging works, update docs temporarily
   echo "Temporary: Use staging endpoint" > TELEMETRY_NOTICE.md
   git commit -am "docs: Temporary telemetry endpoint change"
   git push origin main
   ```

2. **Fix Backend** (timeline depends on issue)

3. **Verify Recovery**:
   ```bash
   # Test from clean install
   cargo install caro --force
   caro "list all files"
   sleep 65
   # Check logs/dashboard for successful upload
   ```
