## Development Environment Setup

### Step 1: Install Rust

```bash
# Install Rust via rustup
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh

# Follow prompts (defaults are fine)
# Restart shell or run:
source $HOME/.cargo/env

# Verify installation
rustc --version  # Should be 1.75.0 or newer
cargo --version
```

---

### Step 2: Clone Repository

```bash
# Clone via HTTPS
git clone https://github.com/username/caro.git
cd caro

# Or via SSH (if you have SSH keys set up)
git clone git@github.com:username/caro.git
cd caro
```

---

### Step 3: Initial Build

```bash
# Build in debug mode (fast, includes debug symbols)
cargo build

# This will:
# 1. Download all dependencies (~150 crates)
# 2. Compile the project
# 3. Create binary at target/debug/caro

# Expected time: 3-5 minutes (first build)
# Subsequent builds: <30 seconds (incremental)
```

**Troubleshooting**:
- If build fails with linker errors (macOS): `xcode-select --install`
- If build fails with OpenSSL errors (Linux): `sudo apt install libssl-dev pkg-config`
- If build fails with permission errors: Check directory permissions

---

### Step 4: Verify Setup

```bash
# Run the binary
./target/debug/caro --version
# Output: caro 1.1.0-beta

# Run tests
cargo test
# All tests should pass

# Run linter
cargo clippy -- -D warnings
# No warnings expected

# Check formatting
cargo fmt --check
# Should exit cleanly (no formatting changes needed)
```

**Success Criteria**: All commands complete without errors

### Key Modules

#### `src/agent/mod.rs`
**Purpose**: Orchestrates command generation workflow

**Key Components**:
- `AgentLoop`: Main agent loop structure
- `generate_command()`: Entry point for command generation
- `repair_command()`: Validation-triggered retry logic
- `refine_command()`: Confidence-based refinement

**When to modify**: Adding new agent loop features, refinement strategies

---

#### `src/backends/`
**Purpose**: Command generation implementations

**Backends**:
- `static_matcher.rs`: Deterministic pattern matching (86.2% accuracy)
- `embedded/`: LLM-based generation (SmolLM, Qwen)

**When to modify**: Adding new patterns, improving LLM prompts, new backends

---

#### `src/safety/patterns.rs`
**Purpose**: Safety validation rules

**Contains**: 75+ safety patterns blocking dangerous commands

**When to modify**: Adding new safety patterns, fixing false positives

**Important**: All changes require tests in `tests/safety_patterns.rs`

---

#### `src/prompts/`
**Purpose**: LLM prompt templates

**Files**:
- `smollm_prompt.rs`: SmolLM-specific prompts
- `qwen_prompt.rs`: Qwen-specific prompts (future)

**When to modify**: Improving generation quality, adding few-shot examples

---

#### `src/validator/`
**Purpose**: Command validation logic

**Checks**:
- Safety (dangerous patterns)
- Platform compatibility (BSD vs GNU)
- Shell compatibility (POSIX compliance)

**When to modify**: Adding new validation rules, platform detection

### Running Tests

**Test Levels**:

**1. Unit Tests** (fast, <10 seconds)
```bash
cargo test --lib
```
- Tests individual functions and modules
- Mock external dependencies
- Run frequently during development

---

**2. Integration Tests** (medium, ~1 minute)
```bash
cargo test --test '*'
```
- Tests end-to-end workflows
- Real backend integration
- Run before committing

---

**3. Safety Tests** (slow, ~5 minutes)
```bash
cargo test --test safety_patterns
```
- Tests all 75 safety patterns
- Critical for safety validation changes
- Run before submitting PRs touching safety code

---

**4. Beta Test Suite** (very slow, ~10 minutes)
```bash
cargo test --test beta_test_suite -- --test-threads=1
```
- Tests all 75 beta test cases
- LLM inference (slow)
- Run before releases

---

**Test Tips**:
```bash
# Run single test
cargo test test_name

# Run tests with output (see println!)
cargo test -- --nocapture

# Run tests in parallel (default)
cargo test

# Run tests serially (useful for debugging)
cargo test -- --test-threads=1

# Show test names without running
cargo test -- --list
```

### Contribution Process

**Step 1: Claim Issue**
- Comment on GitHub issue: "I'd like to work on this"
- Wait for maintainer approval (usually <24 hours)

**Step 2: Understand Scope**
- Read issue description carefully
- Ask clarifying questions in issue comments
- Check related code/tests

**Step 3: Implement**
- Create feature branch: `git checkout -b fix/issue-123`
- Make changes (follow existing patterns)
- Add tests (aim for 100% coverage of new code)
- Update documentation (if user-facing change)

**Step 4: Self-Review**
- Run all checks: `cargo test && cargo clippy && cargo fmt`
- Review your own diff (look for mistakes)
- Check commit message follows conventions

**Step 5: Submit PR**
- Push branch: `git push origin fix/issue-123`
- Create PR with description:
  - What changed
  - Why it changed
  - How to test
  - Screenshots (if UI change)
- Link to issue: "Closes #123"

**Step 6: Address Feedback**
- Respond to review comments
- Make requested changes
- Push updates (no force-push unless requested)
- Mark conversations as resolved

**Step 7: Merge**
- Maintainer will merge after approval
- Celebrate! ðŸŽ‰
- Your name in CONTRIBUTORS.md

    let prompt_lower = prompt.to_lowercase();

    // Example existing pattern
    if prompt_lower.contains("list") && prompt_lower.contains("files") {
        return Some("ls -la".to_string());

**Step 2: Add New Pattern**
```rust
// Add after existing patterns
if (prompt_lower.contains("show") || prompt_lower.contains("list") || prompt_lower.contains("print"))

    && (prompt_lower.contains("variable") || prompt_lower.contains("vars"))
{
    return Some("printenv".to_string());
}
```

## Testing Best Practices

### Writing Good Tests

**1. Test Names Should Be Descriptive**
```rust
// âœ… GOOD
#[test]
fn test_static_matcher_returns_ls_for_list_files_query() { ... }

// âŒ BAD
#[test]
fn test_matcher() { ... }
```

---

**2. Test One Thing**
```rust
// âœ… GOOD: Tests one pattern
#[test]
fn test_disk_usage_pattern() {
    assert_eq!(matcher.match_command("show disk usage"), Some("df -h".into()));
}

// âŒ BAD: Tests multiple unrelated things
#[test]
fn test_everything() {
    assert_eq!(matcher.match_command("show disk usage"), Some("df -h".into()));
    assert_eq!(matcher.match_command("list files"), Some("ls -la".into()));
    assert_eq!(matcher.match_command("kill process"), Some("pkill".into()));
}
```

---

**3. Include Edge Cases**
```rust
#[test]
fn test_safety_validation_edge_cases() {
    let validator = SafetyValidator::new();

    // Normal case
    assert!(validator.validate("ls -la").is_safe());

    // Edge case: Command with semicolon
    assert!(!validator.validate("ls -la; rm -rf /").is_safe());

    // Edge case: Empty command
    assert!(!validator.validate("").is_safe());

    // Edge case: Only whitespace
    assert!(!validator.validate("   ").is_safe());
}
```

---

**4. Use Helper Functions**
```rust
// Helper for common setup
fn create_test_matcher() -> StaticMatcher {
    StaticMatcher::new()
}

#[test]
fn test_pattern_1() {
    let matcher = create_test_matcher();
    // Test...
}

#[test]
fn test_pattern_2() {
    let matcher = create_test_matcher();
    // Test...
}
```

**Public API**: Always document

///

/// * `Ok(GeneratedCommand)` - Successfully generated command
/// * `Err(GeneratorError)` - Failed to generate command
///

/// \```

    // Implementation...

**Internal functions**: Document complex logic
```rust

// a list of known dangerous patterns (rm -rf, dd, etc.)

    // Implementation...
}
```

---

### Error Handling

**Use `Result` for fallible operations**:
```rust
// âœ… GOOD
fn load_config() -> Result<Config, ConfigError> {
    let content = std::fs::read_to_string("config.toml")?;
    let config = toml::from_str(&content)?;
    Ok(config)
}

// âŒ BAD: Using unwrap() (panics on error)
fn load_config() -> Config {
    let content = std::fs::read_to_string("config.toml").unwrap();
    toml::from_str(&content).unwrap()
}
```

**Provide context for errors**:
```rust
use anyhow::{Context, Result};

fn load_config() -> Result<Config> {
    let content = std::fs::read_to_string("config.toml")
        .context("Failed to read config file")?;

    let config = toml::from_str(&content)
        .context("Failed to parse config TOML")?;

    Ok(config)
}
```

## Troubleshooting

### Build Fails

**Error**: "linker `cc` not found"
**Fix** (macOS): `xcode-select --install`
**Fix** (Linux): `sudo apt install build-essential`

---

**Error**: "could not find `libssl`"
**Fix** (Linux): `sudo apt install libssl-dev pkg-config`

---

### Tests Fail

**Error**: Tests pass locally but fail in CI
**Cause**: Platform-specific behavior, race condition
**Fix**: Add `#[cfg(target_os = "...")]` or `#[serial]` attribute

---

**Error**: "test result: FAILED. 0 passed; 1 failed"
**Fix**: Run `cargo test -- --nocapture` to see output, debug failing test

---

### Clippy Warnings

**Warning**: "variable does not need to be mutable"
**Fix**: Remove `mut`: `let x = 5;` instead of `let mut x = 5;`

---

**Warning**: "this expression creates a reference which is immediately dereferenced"
**Fix**: Remove unnecessary `&`: `foo(x)` instead of `foo(&x)`
