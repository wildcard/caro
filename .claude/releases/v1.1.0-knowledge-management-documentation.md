## Purpose

This document defines the knowledge management and documentation strategy for v1.1.0-beta release, ensuring all critical knowledge is captured, organized, and accessible for current team members, future releases, new contributors, and long-term project sustainability.

---

## Audience

- **Primary**: Release Manager, Documentation Lead
- **Secondary**: Full team (contributors and consumers)
- **Tertiary**: Future maintainers, new contributors

---

## Core Principles

1. **Documentation as Code**: Version-controlled, reviewed, tested
2. **Single Source of Truth**: No duplicate or conflicting docs
3. **Progressive Disclosure**: Right level of detail for each audience
4. **Discoverability**: Easy to find what you need
5. **Currency**: Keep docs up-to-date or mark as outdated

### Example ADRs for v1.1.0-beta

**ADR-001: Use Embedded ML Models Over API-Only Backends**

**Context**: Need to decide between embedded models (SmolLM, Qwen) vs API-only (OpenAI, Anthropic) for command generation.

**Decision**: Support both, but prioritize embedded models as default with API as fallback.

**Consequences**:
- âœ… Positive: Works offline, faster, privacy-friendly, no API costs
- âŒ Negative: Larger binary size, slower inference than GPT-4, need model files
- ğŸ”„ Neutral: Can switch backends via config

**Alternatives**:
- API-only: Rejected (requires internet, costs money, privacy concerns)
- Embedded-only: Rejected (quality limitations for complex queries)

---

**ADR-002: Use Static Pattern Matching Before LLM Fallback**

**Context**: LLM inference is slow (450ms p95) compared to static matching (<50ms). Should we always use LLM or optimize with static patterns?

**Decision**: Implement static pattern matcher for common queries, fallback to LLM for complex ones.

**Consequences**:
- âœ… Positive: 10x faster for common queries (68% of traffic), deterministic output, no LLM errors
- âŒ Negative: Maintenance burden (need to update patterns), less flexible than pure LLM
- ğŸ”„ Neutral: Hybrid complexity (two systems to maintain)

**Alternatives**:
- LLM-only: Rejected (too slow for common queries)
- Static-only: Rejected (can't handle complex queries)

## Section 7: Knowledge Sharing

### Documentation Reviews

**Peer Review Process**:
1. Author writes documentation
2. Creates pull request (docs treated like code)
3. Requests review from 1-2 relevant people
4. Reviewers check:
   - Accuracy (technically correct?)
   - Clarity (easy to understand?)
   - Completeness (covers the topic?)
   - Currency (up-to-date?)
5. Author addresses feedback
6. Merge when approved

**Documentation Standards**:
- Use active voice: "Run the command" not "The command is run"
- Use present tense: "Caro generates" not "Caro will generate"
- Use examples: Show, don't just tell
- Use consistent terminology
- Keep sentences <25 words
- Use headings for scannability

### Automated Documentation

**Rustdoc** (API documentation):
```bash
# Generate API docs
cargo doc --no-deps --open

# Publish to docs.rs (on crate publish)
cargo publish
# Automatically available at https://docs.rs/caro
```

**CI/CD Integration**:
```yaml
# .github/workflows/docs.yml
name: Documentation

on: [push, pull_request]

jobs:
  docs:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Check markdown formatting
        run: markdownlint '**/*.md'
      - name: Check links
        run: markdown-link-check README.md
      - name: Generate API docs
        run: cargo doc --no-deps
      - name: Deploy to GitHub Pages
        if: github.ref == 'refs/heads/main'
        uses: peaceiris/actions-gh-pages@v3
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: ./target/doc
```
