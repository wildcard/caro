## Documentation Philosophy

### Core Principles

**1. User-Centric (Not Feature-Centric)**
- Start with user goals: "I want to find large files"
- Not feature lists: "The static matcher has 50 patterns"
- Organize by tasks, not by internal architecture

**2. Progressive Disclosure**
- Getting Started: 5-minute tutorial (minimal)
- How-To Guides: Task-specific instructions (targeted)
- Reference: Complete details (comprehensive)
- Explanations: Deep dives (optional)

**3. Show, Don't Just Tell**
- Every concept includes a working example
- Code snippets are copy-paste ready
- Screenshots/GIFs for visual workflows
- Real commands, not placeholders (`<your-query>`)

**4. Accuracy & Freshness**
- Docs tested on every release
- Breaking changes documented prominently
- Version-specific docs (v1.1.0 vs v1.0.0)
- Deprecated features clearly marked

**5. Accessible & Inclusive**
- Plain language (avoid jargon where possible)
- Define technical terms when first used
- Support multiple learning styles (text, video, examples)
- Internationalization-ready (English first, structure supports i18n)

## Documentation Architecture

### Divio Documentation System

We follow the **Divio documentation system**: 4 documentation types for 4 user needs

| Type | Purpose | When to Use | Format |
|------|---------|-------------|--------|
| **Tutorials** | Learning-oriented | First-time users | Step-by-step, hand-holding |
| **How-To Guides** | Goal-oriented | Experienced users with specific task | Recipe-style, assumes knowledge |
| **Reference** | Information-oriented | Looking up details | Technical, comprehensive, dry |
| **Explanation** | Understanding-oriented | Want to understand "why" | Conceptual, discussion |

## Content Types & Guidelines

### 1. Tutorials (Learning-Oriented)

**Purpose**: Help first-time users learn caro by doing

**Characteristics**:
- Step-by-step (numbered steps)
- Assumes no prior knowledge
- Focuses on successful first experience
- Includes expected output at each step
- Doesn't explain everything (link to Explanation docs)

**Template**:
```markdown
# Tutorial: [Task Name]

**Goal**: By the end of this tutorial, you'll be able to [specific outcome].

**Prerequisites**:
- [Required knowledge, e.g., "Basic command-line familiarity"]
- [Required software, e.g., "Rust 1.70+ installed"]

**Estimated Time**: [X minutes]

## Step 1: [First Step]

[Clear instruction]

```bash
$ command here
```

**Expected Output**:
```
output here
```

## Step 2: [Second Step]

[Clear instruction]

...

## Summary

You've learned:
- [Key takeaway 1]
- [Key takeaway 2]

**Next Steps**:
- [Link to related tutorial]
- [Link to related how-to guide]
```

**Purpose**: Help experienced users accomplish specific tasks

**Purpose**: Provide comprehensive technical details

- Searchable (good headings, ToC)

## Troubleshooting

### Issue: "command not found: caro"

**Cause**: caro binary not in PATH

**Solution**:
```bash
# Add to PATH (add to ~/.bashrc or ~/.zshrc)
export PATH="$HOME/.cargo/bin:$PATH"

# Reload shell
source ~/.bashrc
```

### Issue: "Permission denied"

**Cause**: Binary not executable

**Solution**:
```bash
chmod +x ~/.cargo/bin/caro
```

### Issue: Installation script fails with curl error

**Cause**: Network issue, proxy, or blocked URL

**Solution**: Download binary manually:
1. Go to [Releases](https://github.com/username/caro/releases)
2. Download binary for your platform
3. Extract and move to PATH:
   ```bash
   tar -xzf caro-*.tar.gz
   mv caro ~/.cargo/bin/
   chmod +x ~/.cargo/bin/caro
   ```

caro supports 3 inference backends. Choose based on your needs.

---

## Quick Comparison

| Backend | Speed | Accuracy | Privacy | Offline | Best For |
|---------|-------|----------|---------|---------|----------|
| **Static** | ‚ö° Fastest (<100ms) | üéØ High (patterns) | üîí Perfect | ‚úÖ Yes | Common queries |

---

## Static Matcher Backend

**How It Works**: Pattern matching against 75+ predefined templates

**Pros**:
- ‚ö° Extremely fast (<100ms)
- üéØ 100% accurate for supported patterns
- üîí No data leaves your machine
- üíæ No disk/memory overhead
- ‚úÖ Works offline

**Cons**:
- ‚ùå Limited to predefined patterns
- ‚ùå Cannot handle novel queries
- ‚ùå No learning or adaptation

**When to Use**:
- Common queries (list files, find by name/size/date)
- Need instant response
- Privacy-critical environments
- Offline usage

**Configuration**:
```toml
[backend]
default = "static"
```

**Example Queries**:
```bash
$ caro "list files"                    # ‚úÖ Supported
$ caro "find files larger than 100MB"  # ‚úÖ Supported
$ caro "show disk usage"               # ‚úÖ Supported
$ caro "count lines of code in Python files"  # ‚ùå Not supported (fallback to embedded)
```

---

## Embedded Backend

**How It Works**: Local ML model (SmolLM 135M or Qwen2.5 1.5B) runs on your machine

**Pros**:
- üöÄ Fast (~500ms)
- üéØ Handles complex/novel queries
- üîí No data leaves your machine
- ‚úÖ Works offline
- üìà Better than static for edge cases

**Cons**:
- ‚è±Ô∏è Slower than static (model inference)
- üíæ Larger binary size (~50MB for model)
- üß† Requires RAM (~500MB for Qwen)
- ‚ùì Occasionally hallucinates

**When to Use**:
- Complex queries beyond static patterns
- Privacy-critical environments
- Offline usage
- Custom workflows

**Configuration**:
```toml
[backend]
default = "embedded"

[backend.embedded]
model = "qwen2.5-1.5b"  # or "smollm-135m"
```

**Model Selection**:
- **SmolLM 135M**: Faster (200ms), less accurate (~80%)
- **Qwen2.5 1.5B**: Slower (500ms), more accurate (~90%)

**Example Queries**:
```bash
$ caro --backend embedded "find Python files with TODO comments"
$ caro --backend embedded "show processes using more than 500MB RAM"
```

---

## Ollama Backend

**How It Works**: Connects to Ollama server running LLM (Llama, CodeLlama, Mistral, etc.)

**Pros**:
- üéØ Highest accuracy (GPT-class models)
- üß† Handles very complex queries
- üîß Customizable (any Ollama model)
- üìà Best for rare/novel tasks

**Cons**:
- üêå Slowest (~1-5s depending on model)
- üåê Requires Ollama server running
- ‚ö†Ô∏è Privacy depends on Ollama setup (local = safe, remote = not)
- üíæ Ollama models are large (7-70GB)

**When to Use**:
- Very complex queries
- Custom models for domain-specific tasks
- Already running Ollama for other projects
- Not privacy-critical OR Ollama is local

**Configuration**:
```toml
[backend]
default = "ollama"

[backend.ollama]
url = "http://localhost:11434"  # Local Ollama
model = "codellama:7b"
```

**Setup Ollama**:
```bash
# Install Ollama
curl https://ollama.ai/install.sh | sh

# Pull a model
ollama pull codellama:7b

# Start Ollama server (runs in background)
ollama serve
```

**Example Queries**:
```bash
$ caro --backend ollama "find all JavaScript files that import React and export a default component"
```

## Fallback Chain

caro automatically falls back if a backend fails:

```
Query ‚Üí Static Matcher
          ‚Üì (no match)
        Embedded Backend
          ‚Üì (timeout/error)
        Ollama Backend (if configured)
          ‚Üì (timeout/error)
        Error: "Could not generate command"
```

**Configure Fallback**:
```toml
[backend]
default = "static"
fallback = ["embedded", "ollama"]
```

# Troubleshooting: Backend Errors

Common backend errors and how to fix them.

---

## Embedded Backend: "Out of Memory"

**Symptom**:
```
Error: Backend 'embedded' failed
Caused by: Out of memory: Failed to allocate 1GB tensor
```

**Cause**: Not enough RAM for model

**Solution 1: Use Smaller Model**
```toml
[backend.embedded]
model = "smollm-135m"  # Only 135MB RAM
```

**Solution 2: Switch to Static**
```bash
$ caro --backend static "your query"
```

**Solution 3: Close Other Apps**
Free up RAM by closing browser tabs, IDEs, etc.

---

## Ollama Backend: "Connection Refused"

**Symptom**:
```
Error: Backend 'ollama' failed
Caused by: Connection refused: http://localhost:11434
```

**Cause**: Ollama server not running

**Solution**:
```bash
# Start Ollama server
ollama serve

# In another terminal, try caro again
$ caro --backend ollama "your query"
```

---

## All Backends: "Timeout"

**Symptom**:
```
Error: Backend timeout after 30s
```

**Cause**: Query too complex or system slow

**Solution 1: Increase Timeout**
```toml
[backend]
timeout_seconds = 60  # Default: 30
```

**Solution 2: Simplify Query**
```bash
# ‚ùå Too complex
$ caro "find all JavaScript files that use async/await and have more than 500 lines"

# ‚úÖ Simpler
$ caro "find JavaScript files with more than 500 lines"
```
