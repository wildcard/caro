## Reporting Philosophy

### Core Principles

**1. Transparency**: Share good news and bad news equally, no surprises
**2. Data-Driven**: Metrics over opinions, evidence over assumptions
**3. Actionable**: Every report includes next steps, decisions needed
**4. Appropriate Depth**: Match detail to audience (executive vs team)
**5. Regular Cadence**: Predictable schedule, avoid reporting fatigue

## Stakeholder Map

### Primary Stakeholders

| Stakeholder | Interest | Reporting Needs | Frequency |
|-------------|----------|-----------------|-----------|
| **Executive Team** | Strategic success, market fit | High-level metrics, risks, go/no-go | Weekly summary |
| **Engineering Leads** | Technical quality, performance | Detailed metrics, blockers, capacity | Daily standup |
| **Product Team** | User adoption, feature success | Usage metrics, feedback, roadmap | Weekly deep-dive |
| **Community Team** | User satisfaction, engagement | Community health, support volume | Daily digest |
| **Security Team** | Safety, compliance, incidents | Vulnerability status, incidents | Weekly + ad-hoc |
| **Contributors** | Progress, opportunities | Milestone status, open issues | Async updates |
| **Beta Users** | Product quality, stability | Known issues, upcoming fixes | As needed |

### Information Needs by Stakeholder

**Executives** (WIIFM: "Is this on track?"):
- North Star Metric: Active Weekly Users (target vs actual)
- Release timeline: On schedule / delayed by X days
- Major risks: Top 3 with mitigation status
- Go/No-Go recommendation: Green / Yellow / Red

**Engineering Leads** (WIIFM: "What's blocking us?"):
- Build status: All platforms passing/failing
- Test coverage: Current % vs target
- Performance: Regression status
- Blockers: What needs unblocking today

**Product Team** (WIIFM: "Are users loving it?"):
- Adoption rate: New users per day
- Success rate: % of commands working
- User feedback: Sentiment analysis
- Top requests: Feature/bug priorities

**Community Team** (WIIFM: "How's the community?"):
- Support volume: Tickets/day, response time
- Community sentiment: Positive/neutral/negative
- Engagement: Discord activity, contributions
- Content performance: Views, shares, comments

## Appendix: Metrics Definitions

### Active Weekly Users (AWU)
**Definition**: Unique users who successfully generated at least 1 command in the past 7 days
**Data Source**: Local telemetry (opt-in), GitHub download counts (proxy)
**Calculation**: `COUNT(DISTINCT user_id) WHERE last_command_timestamp >= NOW() - 7 days`

### Command Success Rate
**Definition**: Percentage of generated commands that users execute (vs reject)
**Data Source**: Beta test results, user surveys
**Calculation**: `(commands_executed / commands_generated) * 100`

### Bug Response Time
**Definition**: Time from bug report to first human response
**Data Source**: GitHub Issues, Discord tickets
**Calculation**: `timestamp_first_response - timestamp_bug_reported`

### Community Engagement
**Definition**: Weekly message count in Discord community channels
**Data Source**: Discord analytics
**Calculation**: `SUM(messages) WHERE channel IN (support, general, development) AND timestamp >= NOW() - 7 days`
