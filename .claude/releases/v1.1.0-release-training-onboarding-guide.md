## Onboarding Philosophy

### Core Principles

**1. Learn by Doing**: Hands-on tasks from day one, paired with experienced team members
**2. Progressive Disclosure**: Start with essentials, layer in complexity over time
**3. Documentation First**: Everything is documented, accessible, searchable
**4. Async-Friendly**: Self-paced learning, recorded sessions, written guides
**5. Psychological Safety**: Questions encouraged, mistakes are learning opportunities

### Time to Productivity Targets

| Role | Day 1 | Week 1 | Week 2 |
|------|-------|--------|--------|
| Contributor (Code) | Read docs, setup env | First PR merged | Independent work |
| Contributor (Docs) | Read guidelines | First doc PR | Multiple PRs |
| Community Moderator | Read policies | Shadow veteran | Independent moderation |
| Beta Tester | Install caro, run tests | Submit 5+ bug reports | Write test cases |
| Release Team | Read planning docs | Attend meetings | Own workstream |


- Read `.claude/beta-testing/test-cases.yaml`
- Pick 10 test cases in your expertise area
- Run them manually: `caro "<natural language query>"`

**Shadow Day** (4-8 hours)
- Monitor #support, #bugs, #feature-requests channels
- Observe how experienced moderators respond
- Draft responses (don't send yet), ask for review
- Handle 1-2 simple questions independently (with backup)

**Bug Report Template**:
```markdown
## Bug Report

**Test Case ID**: bt_XXX (if applicable)
**caro Version**: `caro --version`
**Platform**: macOS 14.1 / Ubuntu 22.04 / etc.
**Backend**: static / embedded / ollama

**Natural Language Input**:
"find files larger than 100MB"

**Expected Behavior**:
Should generate: `find . -type f -size +100M`

**Actual Behavior**:
Generated: `find . -type f -size 100M` (missing '+' prefix)

**Impact**: 
Command returns no results even when large files exist.

**Severity**: Medium (incorrect output, but safe)

**Reproducibility**: 100% (every time)

**Additional Context**:
- Only affects size-based queries
- Works correctly for "files smaller than"
```

```bash
# 1. Install Homebrew (if not already installed)
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

# 2. Install Rust
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
source $HOME/.cargo/env

# 3. Install development tools
brew install git
brew install gh  # GitHub CLI

# 4. Configure git
git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"

# 5. Clone repository
git clone https://github.com/anthropics/caro.git
cd caro

# 6. Build and test
cargo build --release
cargo test --all-features

# 7. Install caro locally
cargo install --path .

# 8. Verify
caro --version
caro "list files modified today"
```

**Expected Build Time**: 3-5 minutes on modern hardware

**Troubleshooting**:

| Issue | Solution |
|-------|----------|
| "rustc not found" | Run `source $HOME/.cargo/env` or restart terminal |
| Build fails with link errors | Install Xcode Command Line Tools: `xcode-select --install` |
| Tests fail | Check Rust version: `rustc --version` (need 1.70+) |
| Permission denied | Check directory permissions, may need `sudo` for install |

## Process & Tool Training

### Git Workflow

**Branch Naming Convention**:
```
feat/short-description       # New feature
fix/issue-number-description # Bug fix
docs/description             # Documentation
refactor/description         # Code refactoring
test/description             # Test improvements
```

**Commit Message Format**:
```
type(scope): Brief description (50 chars max)

Detailed explanation of what changed and why (wrap at 72 chars).
Can have multiple paragraphs.

- Bullet points for key changes
- Use imperative mood ("Add feature" not "Added feature")

Fixes #123
Related to #456
```

**Types**: feat, fix, docs, refactor, test, chore, perf, ci

**Pull Request Process**:

1. **Create Branch**:
   ```bash
   git checkout -b feat/your-feature-name
   ```

2. **Make Changes & Commit**:
   ```bash
   git add .
   git commit -m "feat(backend): Add retry logic for model loading"
   ```

3. **Push Branch**:
   ```bash
   git push origin feat/your-feature-name
   ```

4. **Create PR** (via GitHub web UI or `gh` CLI):
   ```bash
   gh pr create --title "Add retry logic for model loading" --body "Fixes #123"
   ```

5. **PR Template** (filled automatically):
   ```markdown
   ## Description
   Brief description of what this PR does.
   
   ## Type of Change
   - [ ] Bug fix
   - [ ] New feature
   - [ ] Breaking change
   - [ ] Documentation update
   
   ## Testing
   - [ ] Unit tests pass
   - [ ] Integration tests pass
   - [ ] Manual testing completed
   
   ## Checklist
   - [ ] Code follows style guidelines
   - [ ] Self-review completed
   - [ ] Comments added for complex logic
   - [ ] Documentation updated
   - [ ] No new warnings
   ```

6. **Address Review Feedback**:
   ```bash
   # Make changes based on feedback
   git add .
   git commit -m "fix: Address review feedback"
   git push origin feat/your-feature-name
   ```

7. **Merge** (after approval):
   - Maintainer will merge using "Squash and merge"
   - Delete branch after merge


2. Read the error message
3. Fix locally and push again
4. CI will automatically re-run

### FAQs

**General**:

Q: How much time should I commit?
A: Varies by role. Contributors: 5-20 hours/week. Release team: 20-40 hours/week. Very flexible, communicate your availability.

Q: I found a bug in the documentation. Should I fix it?
A: Yes! Small fixes don't need approval. Just submit a PR.

Q: Can I work on features not in the v1.1.0 scope?
A: After v1.1.0 launch (Jan 15), yes! Before launch, focus on release-critical work.

Q: I'm stuck. Who do I ask?
A: Ask in #development on Discord. If urgent, tag @Engineering Lead.

**Technical**:

Q: Which Rust version do I need?
A: 1.70 or later. Check with `rustc --version`.

Q: Tests are failing locally but pass in CI. Why?
A: Check for platform-specific behavior. You may be on macOS, CI tests Linux too.

Q: Can I add a new dependency?
A: Needs approval. Read `v1.1.0-release-dependency-management-third-party.md` first.

Q: How do I run tests for a specific module?
A: `cargo test --package caro --lib safety::tests` (example for safety module)

**Process**:

Q: How long until my PR is reviewed?
A: Target: <24 hours for first review. Complex PRs may take 2-3 days.

Q: My PR was approved. Can I merge it?
A: No, maintainers will merge. They may wait to batch changes.

Q: I disagree with review feedback. What do I do?
A: Reply with your reasoning. Reviewers are open to discussion. Escalate to Engineering Lead if needed.

## Appendix: Onboarding Feedback Loop

### New Hire Survey (End of Week 2)

**1. How prepared did you feel on Day 1?** (1-5 scale)
**2. Which documentation was most helpful?**
**3. Which documentation was confusing or missing?**
**4. How responsive was the team to questions?** (1-5 scale)
**5. What would you change about onboarding?**
**6. On a scale of 1-5, how ready do you feel to contribute independently?**

### Continuous Improvement

- Collect feedback after each cohort
- Update onboarding materials based on feedback
- Track time-to-productivity metrics
- Iterate on training content and process
