## Purpose

This document defines the strategy for managing technical debt and maintaining code health throughout the v1.1.0-beta release lifecycle, balancing speed-to-market with long-term maintainability and preventing debt accumulation that could slow future development.

---

## Audience

- **Primary**: Engineering Lead, Engineers
- **Secondary**: Release Manager, Contributors
- **Tertiary**: Future maintainers

---

## Technical Debt Philosophy

### Core Principles

1. **Acknowledge Debt**: Make conscious decisions, document tradeoffs
2. **Strategic Debt**: Some debt is acceptable to ship faster (with plan to repay)
3. **Track Everything**: Maintain debt inventory, prioritize paydown
4. **Continuous Paydown**: Allocate time every sprint for debt reduction
5. **Prevention > Cure**: Avoid creating new debt when possible

### Types of Technical Debt

**Intentional Debt** (Conscious tradeoffs):
- Quick fixes to meet deadlines (marked with TODO/FIXME)
- Simplified implementations (with plan to improve)
- Skipped optimizations (good enough for now)
- Delayed refactoring (code works, but messy)

**Unintentional Debt** (Accumulated over time):
- Outdated dependencies
- Deprecated APIs
- Inconsistent patterns
- Missing tests
- Poor documentation

**Unavoidable Debt** (External factors):
- Breaking changes in dependencies
- Platform changes (OS updates, new Rust versions)
- Security vulnerabilities requiring urgent patches
- Architecture limitations discovered later

## Section 2: Debt During Release (Week 1-4)

### Debt Creation Rules

**When Creating Debt is Acceptable**:
- Hotfix for P0 bug (quality fix now, clean up later)
- Quick workaround for discovered issue (with plan to address root cause)
- Performance optimization deferred (works, just not optimal)
- Refactoring postponed (code messy but functional)

**When Creating Debt is NOT Acceptable**:
- Safety validation shortcuts (NEVER acceptable)
- Privacy violations (NEVER acceptable)
- Security vulnerabilities (NEVER acceptable)
- User-facing bugs (fix properly, not bandaid)
- Breaking changes without migration path

## Section 3: Debt Paydown Strategy

### Allocation Strategy

**Time Allocation**:
- Week 1 (Stabilization): 10% of engineering time for debt paydown
- Week 2-4: 20% of engineering time for debt paydown
- Steady State: 25% of engineering time for debt paydown (1 day per week)

**Prioritization**:
1. **P0 Debt**: Fix immediately (blocks release or critical functionality)
2. **P1 Debt**: Fix within 2 weeks (high impact, manageable effort)
3. **P2 Debt**: Fix within 3 months (medium impact)
4. **P3 Debt**: Fix opportunistically (low impact, large effort)

**Debt Paydown Cadence**:
- Daily: P0 debt (as discovered)
- Weekly: P1 debt (1-2 items per week)
- Monthly: P2 debt (3-5 items per month)
- Quarterly: P3 debt (major refactorings)

---

### Debt Reduction Techniques

**1. Boy Scout Rule**: Leave code better than you found it
```rust
// Before (technical debt):
fn parse_json(s: &str) -> String {
    s.to_string() // Just returns input, no parsing!
}

// After (debt paydown while working nearby):
fn parse_json(s: &str) -> Result<Value, Error> {
    serde_json::from_str(s)
}
```

**2. Incremental Refactoring**: Small improvements over time
```rust
// Week 1: Add types
fn generate(query: &str) -> String { ... }
// Week 2: Add error handling
fn generate(query: &str) -> Result<String, Error> { ... }
// Week 3: Add validation
fn generate(query: &Query) -> Result<Command, Error> { ... }
```

**3. Strangler Fig Pattern**: Gradually replace old code
```rust
// Old system (technical debt)
fn old_command_generator(query: &str) -> String { ... }

// New system (being built)
fn new_command_generator(query: &Query) -> Result<Command, Error> { ... }

// Router (gradually migrate)
fn generate_command(query: &str) -> Result<String, Error> {
    if should_use_new_system(query) {
        new_command_generator(&parse_query(query)?)
            .map(|cmd| cmd.to_string())
    } else {
        Ok(old_command_generator(query))
    }
}
```

**4. Debt Days**: Dedicated time for paydown
- Once per month: Full team "Debt Day"
- Focus on P2/P3 debt that requires coordination
- Celebrate debt reduction (e.g., "Closed 10 debt items!")

## Section 6: Refactoring Strategy

### When to Refactor

**Good Reasons**:
- Code is duplicated (DRY principle violated)
- Code is confusing (readability issue)
- Code is slow (performance issue)
- Code is hard to test (design issue)
- Code violates patterns (inconsistency)

**Bad Reasons**:
- "I don't like the style" (subjective)
- "It could be more clever" (clever is bad)
- "Let me try a new pattern" (consistency matters)
- "Perfection" (good enough is okay)

---

### Refactoring Checklist

**Before Refactoring**:
- [ ] Tests exist for the code (write if missing)
- [ ] Tests pass (baseline)
- [ ] Refactoring is worth the risk (high value)
- [ ] Have time to do it properly (not rushed)

**During Refactoring**:
- [ ] Small, incremental changes
- [ ] Run tests after each change
- [ ] Commit frequently
- [ ] Don't mix refactoring with features

**After Refactoring**:
- [ ] All tests still pass
- [ ] No new warnings
- [ ] Performance not degraded (benchmark)
- [ ] Documentation updated

---

### Safe Refactoring Techniques

**1. Extract Function**:
```rust
// Before: Long function
fn generate_command(query: &str) -> Result<String, Error> {
    // 50 lines of code...
}

// After: Extracted helpers
fn generate_command(query: &str) -> Result<String, Error> {
    let intent = classify_intent(query)?;
    let platform = detect_platform();
    let cmd = match_pattern(intent, platform)?;
    validate_safety(cmd)
}
```

**2. Extract Struct**:
```rust
// Before: Many parameters
fn generate(query: &str, platform: Platform, backend: Backend, config: Config) -> Result<String, Error> { ... }

// After: Struct with builder
struct CommandGenerator {
    platform: Platform,
    backend: Backend,
    config: Config,
}

impl CommandGenerator {
    fn generate(&self, query: &str) -> Result<String, Error> { ... }
}
```

**3. Replace Conditional with Polymorphism**:
```rust
// Before: Big match statement
fn generate(backend: BackendType, query: &str) -> Result<String, Error> {
    match backend {
        BackendType::Static => { /* 20 lines */ },
        BackendType::Embedded => { /* 30 lines */ },
        BackendType::Ollama => { /* 25 lines */ },
    }
}

// After: Trait-based polymorphism
trait Backend {
    fn generate(&self, query: &str) -> Result<String, Error>;
}

struct StaticBackend;
impl Backend for StaticBackend { ... }

struct EmbeddedBackend;
impl Backend for EmbeddedBackend { ... }
```

### Performance Optimization Process

1. **Measure**: Profile to find bottlenecks
2. **Prioritize**: Focus on hot paths (80/20 rule)
3. **Optimize**: Implement improvement
4. **Measure Again**: Verify improvement
