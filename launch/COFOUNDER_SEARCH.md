## YC Co-Founder Matching Profile

### About the Opportunity

**We're building GitHub Copilot for the terminal.**

cmdai is an open-source Rust CLI that converts natural language to safe shell commands using local AI. Think "deploy to AWS" â†’ correct kubectl/docker commands, instantly.

**The vision:** AI-native operations platform trusted by 10,000+ engineering teams by 2028. Following the PostHog playbook: open-source core + cloud/enterprise SaaS.

**Current traction:**
- Working MVP with Apple Silicon optimization (MLX)
- Comprehensive safety validation system
- 44 tests passing, production-ready architecture
- Active development and clear roadmap to V1.0
- AGPL-3.0 licensed, building in public

**The business model:**
- Free: Open-source CLI (growth engine)
- Pro: $10/user/mo (power users)
- Team: $20/user/mo (5-50 person teams)
- Enterprise: $50-75/user/mo (compliance, audit logs, SSO, RBAC)

**Path to $50M ARR:**
- Q1 2025: Cloud launch, $2K MRR
- Q4 2025: $100K MRR, Series A raise

- PostHog: $40M ARR in 3 years (same model)
- GitLab: $11B IPO (open source + enterprise)
- Supabase: $100M ARR in 3 years (same model)

**Core requirements:**
- 7+ years backend/systems engineering experience
- Deep technical expertise (distributed systems, databases, infrastructure)
- Rust experience or strong eagerness to learn (we'll support the ramp-up)
- Led engineering teams (5+ people minimum)
- Startup experience strongly preferred (especially YC alum or ex-FAANG)

**What you'll own:**

- Cloud platform scaling (1K â†’ 100K â†’ 1M users)

- Infrastructure strategy (security, performance, reliability)
- Technical credibility with VCs and enterprise customers

**What excites you:**
- Open source + SaaS hybrid model
- Developer tools and DevOps space
- AI/LLM applications in production
- Building something from scratch with massive potential
- PostHog/GitLab/Supabase style growth trajectory

**Commitment:**
- Full-time (this is a founder role, not a side project)
- Available to start within 1-3 months
- Based anywhere (we're remote-first)
- Aligned on multi-year journey (this is a 5-7 year commitment)

## Vetting Questionnaire (20 Questions)

Use these during your interview process to assess technical fit, cultural fit, and commitment.

### Technical Fit (Architecture, Rust, Scaling)

**1. Architecture & Systems Design**

*"Walk me through how you'd architect cmdai's cloud backend to handle 100,000 concurrent users generating commands. What would your stack look like?"*

**What you're listening for:**
- Experience with distributed systems
- Understanding of load balancing, caching, queueing
- Database choices (PostgreSQL, Redis, etc.)
- API design thinking
- Scalability mindset

**Red flag:** Vague answers, no specific technology choices, unrealistic scaling approaches

**Green flag:** Specific stack recommendations with tradeoffs explained, experience with similar scale

---

**2. Rust Expertise**

*"What's your experience with Rust? If limited, what draws you to learning it deeply for this project?"*

**What you're listening for:**
- Current Rust proficiency level (honest self-assessment)
- If not experienced: genuine interest + learning plan
- Understanding of Rust's value prop (safety, performance)
- Experience with similar systems languages (C++, Go)

**Red flag:** "I'll figure it out" without concrete plan, no interest in Rust's philosophy

**Green flag:** Either Rust experience OR concrete plan to ramp up + excitement about memory safety

---

**3. Infrastructure & DevOps**

*"Describe the most complex infrastructure you've built or managed. What were the biggest challenges?"*

**What you're listening for:**
- Hands-on infrastructure experience
- Understanding of CI/CD, monitoring, observability
- Experience with cloud providers (AWS, GCP, Azure)
- Problem-solving approach to infrastructure issues

**Red flag:** Only managed infrastructure, never built it

**Green flag:** Built infrastructure from scratch, scaled it, learned from failures

---

**4. Security & Safety**

*"cmdai has a comprehensive safety validation system to prevent dangerous commands. How would you approach building the cloud version's security layer?"*

**What you're listening for:**
- Security-first mindset
- Understanding of auth, authorization, rate limiting
- Input validation and sanitization thinking
- Awareness of common vulnerabilities

**Red flag:** "We can add security later" mentality

**Green flag:** Security as first-class concern, specific strategies mentioned

---

**5. Performance & Optimization**

*"Our goal is <1s command generation latency at scale. How would you approach achieving and maintaining this?"*

**What you're listening for:**
- Performance engineering experience
- Understanding of profiling, benchmarking
- Optimization strategies (caching, CDN, edge computing)
- Experience with LLM inference optimization

**Red flag:** "Just throw more servers at it"

**Green flag:** Systematic approach to performance, experience with sub-second SLA

---

- Ability to convince and align team
- Willingness to change mind with new data
- Conflict resolution skills

**Red flag:** "I just told them to do it my way"

**Green flag:** Collaborative process, team bought in, or changed approach based on feedback

- Investment in team growth
- Concrete mentorship examples

- Track record of developing talent

- **0 points:** Weak answer, red flag

**Scoring guide:**
- **35-40:** Excellent fit, strong candidate
- **28-34:** Good fit, worth deeper exploration
- **20-27:** Uncertain fit, proceed with caution

**Critical failures:** Any of these are automatic disqualifiers regardless of score:

- Fundamentally opposed to open source model
- Values deeply misaligned
- Unrealistic equity expectations with no flexibility
- No technical leadership experience

## Trial Project (1-2 Week Collaboration)

**Purpose:** Work together on a real feature before making the co-founder decision. This is like dating before marriage.

### Why a Trial?

**For you:**
- See their code quality and technical judgment
- Assess collaboration and communication style
- Observe problem-solving approach
- Test cultural fit in real work

**For them:**
- Experience the codebase and technical challenges
- See if they enjoy the work and problem space
- Assess your working relationship
- Evaluate the opportunity hands-on

### Trial Structure

**Duration:** 1-2 weeks (can be part-time if they're still employed)

**Compensation:**
- Option 1: Unpaid (sweat equity if they join)
- Option 2: Pay consulting rate ($200-300/hr) for their time
- Recommended: Option 2 if they need to take time off work

**Commitment:**
- 10-20 hours of focused work
- Daily async updates
- 2-3 pair programming sessions
- Code reviews on each other's work

### Recommended Feature: Cloud API Authentication Foundation

**Why this feature:**
- Core to the cloud platform
- Requires architectural thinking
- Demonstrates security mindedness
- Involves Rust backend work
- Scoped but meaningful

**The task:**

```
Design and implement the authentication foundation for cmdai's cloud API.

Requirements:
1. JWT-based authentication system
2. User registration and login endpoints
3. Token refresh mechanism
4. Rate limiting (prevent abuse)
5. Input validation and security best practices
6. Comprehensive error handling
7. Unit and integration tests

Tech stack:
- Rust + Axum (web framework)
- PostgreSQL (user storage)
- Redis (token storage, rate limiting)

Deliverables:
- Working code with tests
- API documentation (OpenAPI spec)
- Architecture decision record (ADR) explaining design choices
- Security considerations document

Time estimate: 10-20 hours
```

**How to structure it:**

**Week 1:**

**Day 1 (2 hours):**
- Kickoff call: Review requirements together
- Discuss architectural approach
- Align on tech choices
- Set up their dev environment

**Day 2-3 (4-6 hours):**
- They design the system (architecture doc)
- Async review and feedback from you
- Pair programming session (1 hour) to align

**Day 4-5 (4-6 hours):**
- Implementation phase
- They submit code for review
- You review and provide feedback

**Week 2 (if needed):**

**Day 6-7 (2-4 hours):**
- Address review feedback
- Refine implementation
- Add tests and documentation

**Day 8 (1 hour):**
- Final review session
- Retrospective: What went well? What was challenging?
- Discuss learnings and fit

### What to Observe

**Technical:**
- [ ] Code quality (readable, maintainable, idiomatic Rust)
- [ ] Security awareness (input validation, auth best practices)
- [ ] Testing approach (comprehensive, meaningful tests)
- [ ] Documentation quality (clear, thorough)
- [ ] Architectural thinking (good design choices with rationale)

**Collaboration:**
- [ ] Communication style (clear, proactive, responsive)
- [ ] Receptiveness to feedback (non-defensive, iterative)
- [ ] Question-asking (asks good questions when uncertain)
- [ ] Autonomy vs. guidance (can work independently but knows when to ask)

**Cultural:**
- [ ] Excitement about the problem (engaged, interested)
- [ ] Attention to detail (thoroughness, quality-minded)
- [ ] Work style compatibility (async communication, pace)
- [ ] Problem-solving approach (systematic, thoughtful)

### Decision Framework

**After the trial, ask yourself:**

1. **Would I want to work with this person for the next 5 years?**
   - If hesitation, that's your answer

2. **Did they produce high-quality work?**
   - Code quality, tests, documentation all matter

3. **Do they communicate well?**
   - Async updates, clear explanations, responsive

4. **Are they excited about cmdai?**
   - Genuine interest, asked good questions, showed enthusiasm

5. **Do our work styles mesh?**
   - Pace, communication, decision-making all compatible

**If 4+ are "yes" â†’ Strong candidate, move forward**

**If 2-3 are "yes" â†’ Uncertain, consider longer trial or pass**

**If 0-1 are "yes" â†’ Not the right fit**

### Alternative Trial Projects

**Option 2: Safety Pattern Expansion**
- Add 20 new dangerous command patterns

- Why: Shows attention to detail, security mindset
